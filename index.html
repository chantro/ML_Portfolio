🚨 핵심 질문

모델링 과정에서 가장 먼저 점검한 질문: "학습 데이터의 모든 특성이 서비스 환경에서도 동일하게 수집 가능한가?"

✅ 접근 방향

feature를 줄여도 Recall을 유지할 수 있다면, 실제 배포 가능한 모델이 된다. Threshold 조정으로 Precision-Recall 트레이드오프를 실험적으로 검증.

🔬
Feature 설계의 현실성

연구 데이터의 모든 feature가 실시간 서비스에서 얻을 수 있는 것은 아님. 배포 가능성을 고려한 feature selection이 모델 성능만큼 중요하다는 것을 체감.

⚖️
Threshold는 설계 결정

threshold는 단순 튜닝 파라미터가 아니라 "사기를 놓치는 것 vs 정상을 차단하는 것" 중 어떤 비용이 더 큰지를 결정하는 도메인 판단. 모델 저장 시 함께 래핑하여 일관성 확보.

🏗️
Ablation의 가치

feature를 줄였을 때 성능이 오히려 오른 결과는 ablation 실험이 없었다면 놓쳤을 인사이트. 가설을 세우고 대조 실험으로 검증하는 습관이 ML 설계의 핵심임을 확인.

① 협업 필터링 — BPR
Bayesian Personalized Ranking

사용자의 조회 이력과 스크랩 이력을 함께 implicit feedback으로 활용해 BPR 협업필터링을 설계했습니다. 관찰된 공지와 미관찰 공지의 선호 순서를 학습해 사용자별 Top-5 추천을 생성했습니다.
※ BPR 모델은 기본적으로 binary interaction을 기반으로 하기 때문에, 조회와 스크랩을 동일한 상호작용으로 처리했습니다. ※ 이미 조회한 공지는 최종 추천 결과에서 다시 제외해 중복 노출을 줄였습니다.

→
입력: viewedPosts + scrappedPosts 기반 User-Item 상호작용 데이터
→
출력: 사용자별 Top-5 추천 공지 → Firestore 저장
→
평가: 상호작용 이력이 존재하는 사용자 기준 오프라인 평가에서 Top-5 precision 약 60% 수준 확인

스크랩 공지 기반 유사 공지 추천

키워드 빈도 기반 추천의 문맥 반영 한계를 인식, 임베딩 기반 접근을 고려했다. 그러나 공지 데이터 규모(~1,200개)가 작아 과적합 가능성이 높고, 텍스트도 짧아 복잡한 모델보다 단순한 TF-IDF가 이 도메인에 더 적합하다고 판단, 채택하지 않음.

💡 데이터 규모·텍스트 길이 분석 → TF-IDF가 이 도메인에 더 적합
최종 채택 — BPR + TF-IDF 하이브리드
소규모 환경에 맞는 경량 하이브리드

조회·스크랩 기반 BPR은 행동 데이터가 쌓인 사용자에게 개인화 추천을 제공합니다. TF-IDF 콘텐츠 기반은 상호작용 데이터가 부족한 신규 사용자에게도 즉시 작동하는 보완 추천입니다. 두 전략의 결합으로 데이터 희소성 문제를 구조적으로 대응.

✅ 조회·스크랩 기반 BPR + 최근에 스크랩한 공지를 기준으로 한 TF-IDF · 신규 사용자 문제 구조적 완화
추천 구조와 역할 분리
희소한 사용자-공지 구조
관찰된 상호작용 상호작용 없음

사용자 수와 상호작용 수가 충분하지 않은 환경에서는 사용자-공지 행렬이 희소해지므로, 협업필터링만으로는 모든 사용자에게 안정적인 추천을 제공하기 어렵습니다.

🧠
Implicit Feedback 설계

명시적 평점이 없는 환경에서 조회와 스크랩 같은 사용자 행동을 implicit feedback으로 활용했습니다. 어떤 행동을 신호로 정의할지 결정하는 것 자체가 추천 시스템 설계의 핵심임을 체감했습니다.

🔄
구조로 희소성 대응

데이터 부족 문제를 알고리즘으로만 해결하려 하지 않고, BPR(데이터가 있는 사용자)과 TF-IDF(데이터가 부족한 사용자)의 역할을 분리하는 구조 설계로 접근. "언제 어떤 모듈이 작동하는가"를 명확히 정의하는 것이 하이브리드의 핵심.

📋
평가 방법론의 명시

온라인 A/B 평가 없이 오프라인 평가로 검증한 한계를 명시. "측정하지 못한 것을 주장하지 않는다"는 태도가 연구 포트폴리오에서 신뢰를 만든다.
